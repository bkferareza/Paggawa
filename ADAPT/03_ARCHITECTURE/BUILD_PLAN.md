# Build Plan

STATUS: BUILD_PLAN_READY

## 1. Build Strategy

Paggawa v0.1 will be built one lane at a time as two app surfaces under one shared product system. The first objective is to prove the local work loop with a maintainable client-side prototype, not to create production infrastructure.

The build must follow the existing ADAPT lanes exactly. Each lane should produce only its own behavior, pass its own validation criteria, and avoid pulling future-lane functionality forward. The prototype should start with surface-based view state before adding a routing library.

Application source code is not generated by this build-plan correction.

## 2. Confirmed Prototype Stack

- React + Vite + TypeScript
- Paggawa Mobile: phone/mobile-first web/PWA surface for Resident and Worker experiences
- Paggawa Quest: barangay hall-friendly web surface for Barangay Staff assisted-access workflows
- Simple CSS or Tailwind candidate, decided at app scaffold time
- Mock seed data for the first runnable experience
- LocalStorage persistence once user-created or changed records exist
- One workspace and one codebase with surface-specific UI modules
- One shared domain model, mock/local data layer, validation model, and ADAPT delivery system
- No backend
- No native mobile
- No production authentication
- No production database

This direction is confirmed because it keeps the prototype maintainable for a solo developer using VS Code on a personal PC while preserving the required separation between Paggawa Mobile and Paggawa Quest.

## 7. Lane-by-Lane Validation Criteria

### LANE-01-APP-SHELL

- App loads successfully.
- User can switch between Paggawa Mobile and Paggawa Quest surfaces.
- In Paggawa Mobile, user can switch between Resident and Worker experiences.
- Paggawa Mobile Resident dashboard is visible.
- Paggawa Mobile Worker dashboard is visible.
- Paggawa Quest workspace/dashboard is visible.
- Mock seed data is visible from the shared mock data layer.
- No job creation, matching, reputation, backend, auth, payments, GPS, or chat exists.

### LANE-02-JOB-REQUESTS

- Resident can create a job request in Paggawa Mobile.
- Barangay staff can create a job request on behalf of a resident in Paggawa Quest.
- Created job appears in resident view.
- Created job appears in worker nearby jobs view.
- Created job appears in Paggawa Quest quest board.
- Exact address and contact details remain hidden from public discovery views.

### LANE-03-WORKER-DISCOVERY

- Resident can see nearby workers in Paggawa Mobile.
- Resident can view a worker profile.
- Worker profile shows skills, service area, and trust signals.
- Barangay staff can view worker registry in Paggawa Quest.
- Trust copy does not imply guaranteed quality.

### LANE-04-BARANGAY-QUEST-BOARD

- Barangay staff can view open jobs in Paggawa Quest.
- Barangay staff can create assisted requests.
- Barangay staff can view registered workers.
- Quest board clearly shows open job requests.
- Feedback or complaint note placeholder is present without becoming full dispute management.

### LANE-05-MATCHING

- Worker can respond to an open job.
- Resident can view worker response.
- Resident can accept one worker.
- Job status changes to matched.
- Exact address and contact details are not exposed before match.
- Coordination details unlock only for the confirmed match view.

### LANE-06-REPUTATION

- Matched job can be marked completed.
- Resident can leave review.
- Worker completed-job count updates.
- Worker rating appears on profile.
- Reputation display remains simple and avoids overclaiming quality.

## 8. Data and Mocking Plan

Use a small barangay-scale seed dataset with stable IDs. Seed records should cover residents, workers, barangay staff, worker profiles, job requests, responses, matches, reviews, complaint-note placeholders, and one barangay area.

Lane 01 should display seed data only from the shared mock data layer. Lane 02 should introduce LocalStorage persistence for created job requests. Later lanes should extend the same local store for worker responses, matches, reviews, and notes.

The data shape should follow the Domain Model:

- User
- WorkerProfile
- JobRequest
- JobResponse
- Match
- Barangay
- Review
- ComplaintNote

Exact address and contact fields must be stored separately from approximate discovery fields. Discovery views should use barangay, area label, approximate distance, category, status, and service radius rather than GPS coordinates.

## 9. Privacy and Trust Guardrails

- Show approximate location before match.
- Hide exact resident address before match.
- Hide phone number and direct contact details before match.
- Unlock coordination details only after a confirmed match and only for the matched resident and worker.
- Keep quest board items focused on category, status, approximate area, and assisted-request context.
- Barangay staff may assist with private details, but public discovery views must still preserve privacy.
- Use approved trust labels: Barangay-registered, Identity checked, Community-referred, Completed jobs, Rating, Reviews, Referral count, No unresolved complaints.
- Do not use labels that imply guaranteed quality, official certification of skill, or barangay-guaranteed work outcomes.

## 10. What Must Not Be Built Yet

Do not build any of these in v0.1:

- payments
- escrow
- real GPS tracking
- full chat
- official LGU integration
- complex dispute resolution
- advanced AI matching
- full native mobile application
- production authentication
- production database
- nationwide deployment logic
- legal certification of worker quality
- claims that barangay registration guarantees work quality

Do not build future-lane behavior before its lane begins. In particular, Lane 01 must stay limited to the app scaffold, shared product shell, surface selector, Paggawa Mobile resident/worker dashboards, Paggawa Quest workspace/dashboard, navigation, and seed data display.

# ADAPT Correction: Two App Surfaces

Paggawa v0.1 should be built as one maintainable prototype system with two conceptually and structurally separate UI surfaces:

1. Paggawa Mobile
   - Resident experience
   - Worker experience
   - Optimized for phone/mobile usage

2. Paggawa Quest
   - Barangay Staff assisted-access experience
   - Optimized for barangay hall workflows
   - Tablet/kiosk/laptop/desktop-friendly

The product should continue to use one shared domain model, one shared mock/local data layer, one shared state approach, and one shared validation model. This correction does not create two repositories or two independent systems.

paggawa/
├── apps/

│   │   ├── resident/
│   │   └── worker/

For v0.1, the simplest implementation may still be one Vite app project to avoid npm workspace complexity:

paggawa-app/
└── src/
    ├── apps/
    │   ├── mobile/
    │   │   ├── resident/

    │       └── barangay/
    └── shared/
        ├── domain/
        ├── data/
        ├── state/

The build plan must preserve:

- One app project for v0.1 if that is simplest.
- Two app surfaces in UI organization.
- Shared domain/data/state.
- No backend.
- No native mobile.
- No production auth.
- No payments.
- No GPS.
- No chat.

This correction supersedes any earlier single generic app model or three-equal-role mobile model in this document.
